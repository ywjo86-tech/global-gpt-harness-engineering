Purpose: To define the final feature development policy for the Global Agent, including Codex Loop Engineering, hierarchical development planning, UI pre-visualization, step-level QA, final deployment UI confidence verification, progress handoff, and token-efficient development control.

============================================================
1. 최종 판단 요약
============================================================

이번 최종 보고서의 핵심은 전역 에이전트가 신규 프로젝트를 수행할 때 단순히 요구사항을 바로 코드로 구현하지 않고, 먼저 충분한 분석과 추론, 예측을 통해 계층형 개발계획서를 수립하도록 만드는 것이다.

가장 적정한 구조는 다음과 같다.

1. Main Development Plan
2. Step Detail Plans
3. Optional Nested Detail Plans
4. Step Handoff Documents
5. Final Deployment Handoff

이 구조는 두 가지 목적을 동시에 달성한다.

첫째, 프로젝트 초기에 충분히 깊은 계획을 수립하여 산출물의 완성도를 높인다.

둘째, 모든 내용을 하나의 긴 문서에 몰아넣지 않고 단계별 세부계획서로 분리하여 Codex의 토큰 사용량을 줄인다.

최종 운영 방식은 다음과 같다.

- 사용자가 신규 프로젝트 요구사항을 제시한다.
- Codex는 먼저 요구사항을 분석하고 추론한다.
- Codex는 메인계획서를 만든다.
- 메인계획서의 각 단계는 별도 세부계획서와 연결한다.
- 세부계획서가 복잡하면 하위 세부계획서를 만든다.
- Codex는 메인계획서를 기준으로 전체 방향을 유지한다.
- 실제 개발 시에는 현재 단계와 연결된 세부계획서만 검토한다.
- 각 단계 완료 후 핸드오프 문서를 작성한다.
- 다음 단계는 연결된 다음 세부계획서를 기준으로 진행한다.
- 최종 단계에서는 최종 배포용 산출물과 최종 핸드오프를 작성한다.

============================================================
2. 사용자의 요구사항 분석
============================================================

사용자의 요구사항은 다음과 같이 해석된다.

명시 요구사항:

1. 신규 프로젝트 진행 시 요구사항을 제시하면 Codex가 개발계획서를 수립해야 한다.
2. 개발계획서는 충분한 분석, 추론, 예측을 통해 세부적으로 수립되어야 한다.
3. 하지만 문서가 너무 길어지면 토큰 낭비가 발생한다.
4. 따라서 개발계획서는 메인계획서와 단계별 세부계획서로 나누어야 한다.
5. 메인계획서와 세부계획서는 별도 파일로 생성하되 서로 연결되어야 한다.
6. Codex는 메인계획서를 기준으로 전체 개발을 진행해야 한다.
7. 각 단계별로 연결된 세부계획서를 검토해야 한다.
8. 각 단계 완료 후 단계별 핸드오프를 진행해야 한다.
9. 다음 단계는 다음 세부계획서를 기준으로 진행해야 한다.
10. 최종 배포용 산출물과 최종 핸드오프를 진행해야 한다.
11. 세부계획서가 복잡하면 세부계획서 안에 하위 세부계획서를 만들 수 있어야 한다.

추론된 요구사항:

1. 사용자는 코딩 지식이 많지 않으므로 Codex가 계획을 충분히 세워야 한다.
2. 사용자의 중간 개입을 줄이기 위해 초기 계획의 완성도가 높아야 한다.
3. 하지만 계획 문서가 너무 크면 Codex 작업 비용이 커진다.
4. 따라서 문서의 깊이와 토큰 효율을 동시에 만족해야 한다.
5. 개발 중 새로운 아이디어가 생기더라도 즉시 구현하지 않고 범위 확장 후보로 관리해야 한다.
6. 단계별 산출물과 핸드오프가 연결되어야 진행상황을 추적할 수 있다.
7. 최종 산출물은 메인계획서의 목표와 단계별 세부계획서의 결과를 모두 반영해야 한다.

최종 목적:

프로젝트 구성 및 개발계획 단계에서 프로젝트를 충분히 세분화하고, 산출물의 완성도를 높이며, 프로젝트 진행 중 사용자의 반복 개입을 최소화하고, 개발방향의 변경 및 불필요한 범위 확장을 방지하는 것이다.

3. Global Agent 내 기능 위치
============================================================

이 기능은 개별 프로젝트의 기능이 아니라 전역 에이전트의 프로젝트 운영 정책이다.

위치는 다음과 같이 설정한다.

Global Agent
  └─ Planning Policy
      ├─ Hierarchical Development Planning
      ├─ Main Development Plan Policy
      ├─ Step Detail Plan Policy
      ├─ Nested Detail Plan Policy
      ├─ Step Handoff Policy
      └─ Final Deployment Handoff Policy

역할 구분:

| 구분 | 역할 |
|---|---|
| Global Agent | 계층형 계획 정책 원본 보유 |
| Project Main Agent | 메인계획서를 기준으로 프로젝트 운영 |
| Subagents | 각 단계 세부계획서 기준으로 실행 |
| QA Subagent | 단계별 QA 및 최종 QA 검수 |
| Logger/Handoff Subagent | 단계별 핸드오프와 최종 핸드오프 기록 |

중요 원칙:

- 메인계획서는 전체 방향을 관리한다.
- 세부계획서는 단계별 실행을 관리한다.
- 하위 세부계획서는 복잡한 세부 기능을 관리한다.
- Codex는 현재 단계에 필요한 문서만 읽어야 한다.
- 모든 세부계획서는 PLAN_INDEX.md로 연결되어야 한다.

7. Nested Detail Plan 작성 정책
============================================================

세부계획서 안에서도 내용이 복잡해지는 경우 Nested Detail Plan을 생성할 수 있다.

Nested Detail Plan은 다음 경우에 생성한다.

- 하나의 단계가 여러 하위 기능으로 나뉘는 경우
- UI, 백엔드, 테스트, 배포가 한 단계 안에 혼재되는 경우
- 기술적 리스크가 높은 기능이 포함된 경우
- 예상 작업량이 커서 하나의 Step Detail Plan이 길어지는 경우
- Codex가 한 번에 읽고 수행하기 어려울 정도로 문서가 커지는 경우
- 하위 기능별로 별도 QA가 필요한 경우

Nested Plan 파일명 예시:

STEP_02_01_DETAIL_PLAN.md
STEP_02_02_DETAIL_PLAN.md
STEP_03_01_UI_DETAIL_PLAN.md
STEP_03_02_BACKEND_DETAIL_PLAN.md
STEP_04_01_TEST_DETAIL_PLAN.md

Nested Detail Plan 생성 기준:

| 조건 | 처리 |
|---|---|
| 세부계획서가 너무 길어짐 | Nested Plan 생성 |
| 한 단계 안에 독립 기능이 2개 이상 | Nested Plan 생성 |
| 기술 리스크가 큼 | Nested Plan 생성 |
| UI와 기능 구현이 분리 가능 | Nested Plan 생성 |
| 별도 QA가 필요 | Nested Plan 생성 |

중요 원칙:

- Nested Plan은 필요할 때만 생성한다.
- 모든 Nested Plan은 상위 Step Detail Plan에 연결되어야 한다.
- Nested Plan도 1사이클 1수정 원칙을 따른다.
- Nested Plan 완료 후 상위 Step Detail Plan에 결과를 반영한다.

8. PLAN_INDEX.md 연결 정책
============================================================

PLAN_INDEX.md는 메인계획서와 세부계획서의 연결 상태를 관리하는 인덱스 문서이다.

Codex는 개발을 시작하기 전에 PLAN_INDEX.md를 확인하여 현재 단계에 필요한 문서만 읽는다.

- Active Detail Plan:
- Active Nested Plan:
- Current Loop:

## Final Deployment
- HANDOFF_FINAL_DEPLOYMENT.md

Codex 운영 규칙:

1. 다음 작업을 시작하기 전 PLAN_INDEX.md를 먼저 확인한다.
2. 현재 단계와 연결된 Step Detail Plan만 검토한다.
3. 필요할 경우에만 Nested Detail Plan을 검토한다.
4. 완료된 단계는 전체 내용을 반복해서 읽지 않고 Handoff 요약만 참조한다.
5. 단계 완료 후 PLAN_INDEX.md 상태를 업데이트한다.

============================================================
9. Codex 개발 진행 방식
============================================================

Codex는 메인계획서를 기준으로 전체 개발을 진행한다.

그러나 실제 개발 단계에서는 아래 순서로 필요한 문서만 검토한다.

1. PLAN_INDEX.md 확인
2. MAIN_DEVELOPMENT_PLAN.md에서 현재 Phase 확인
3. 현재 Phase와 연결된 Step Detail Plan 확인
4. 필요한 경우 Nested Detail Plan 확인
5. 해당 단계의 Loop 실행
6. 검증 및 QA 수행
7. HANDOFF_STEP_NN.md 작성
8. PLAN_INDEX.md 상태 업데이트
9. 다음 Phase로 이동
10. 모든 Phase 완료 후 HANDOFF_FINAL_DEPLOYMENT.md 작성

중요 규칙:

- Codex는 매 루프마다 전체 개발계획서를 반복 출력하지 않는다.
- Codex는 현재 단계에 필요한 계획서만 참조한다.
- 이전 단계의 상세 내용은 Handoff 요약만 참조한다.
- 다음 단계로 넘어가기 전 현재 단계의 Handoff가 완료되어야 한다.

============================================================
10. 사용자 개입 최소화 정책
============================================================

프로젝트 초기에 충분히 분석한 개발계획서를 수립함으로써 사용자의 반복 개입을 줄인다.

Codex는 아래 항목을 계획 단계에서 최대한 예측하여 문서화한다.

- 요구사항 해석
- 예상 기능 목록
- 제외 범위
- 개발 단계
- 단계별 산출물
- 단계별 검증 방법
- UI 예상도
- 예상 리스크
- 범위 확장 가능성
- 사용자 승인 필요 지점

사용자 개입이 필요한 경우는 아래로 제한한다.

1. 메인계획서 최초 승인
2. UI 예상도 승인
3. 평가 기준 승인
4. 범위 변경 승인
5. 실패 루프가 2회 이상 반복될 위험이 있는 경우
6. 최종 배포 승인

그 외에는 Codex가 계획서와 세부계획서 기준으로 자율 진행한다.

12. 토큰 절약 정책
============================================================

계층형 개발계획서는 토큰을 많이 쓰기 위한 구조가 아니라, 오히려 토큰을 절약하기 위한 구조이다.

토큰 절약 원칙:

1. 메인계획서는 전체 방향만 담는다.
2. 세부 구현 내용은 Step Detail Plan에 분리한다.
3. Codex는 현재 단계에 필요한 세부계획서만 읽는다.
4. 완료된 단계의 세부계획서는 요약만 참조한다.
5. 전체 계획서를 매 루프마다 반복 출력하지 않는다.
6. 핸드오프는 수치와 결정사항 중심으로 작성한다.
7. 변경 없는 영역은 "변경 없음"으로 기록한다.
8. Nested Plan은 필요할 때만 생성한다.
9. 불필요한 장문 설명 대신 표와 체크리스트를 사용한다.
10. 다음 단계 진행 시 PLAN_INDEX.md를 통해 연결 문서만 확인한다.

토큰 낭비를 유발하는 방식:

- 모든 세부 내용을 하나의 문서에 넣는 방식
- 매 단계마다 전체 계획서를 다시 출력하는 방식
- 완료된 단계의 세부 내용을 반복 분석하는 방식
- 현재 단계와 무관한 세부계획서까지 모두 읽는 방식
- 새로운 아이디어를 즉시 개발 범위에 넣는 방식

가장 적정한 방식:

- 전체 방향은 메인계획서
- 실행 내용은 단계별 세부계획서
- 복잡한 기능은 하위 세부계획서
- 현재 작업은 현재 연결 문서만 참조
- 완료 내용은 핸드오프 요약으로 참조

13. UI QA 정책
============================================================

UI 예상도는 단순 참고자료가 아니라 개발 전 사용자와 합의한 목표 화면이다.

단, UI Match Confidence 95% 기준은 다음과 같이 나누어 적용한다.

단계별 UI 개발:

- 대상: 이번 단계에서 개발한 UI 영역, 컴포넌트, 화면 일부
- 기준: 해당 부분 기준 Part-level UI Match Confidence 95% 이상
- 목적: 단계별 품질 확보
- 토큰 절약 방식: 변경된 부분만 비교하고 기록

최종 배포 UI QA:

- 대상: 전체 화면, 전체 사용자 흐름, 전체 UI 구성
- 기준: Final UI Match Confidence 95% 이상
- 목적: 사용자 목표 화면과 최종 결과물 일치 확인
- 수행 시점: 최종 배포용 핸드오프 문서 작성 시

중요 원칙:

- 매 단계마다 전체 UI를 95% 비교하지 않는다.
- 단계별로는 변경된 UI 부분만 95% 검수한다.
- 전체 UI 95% 검수는 최종 배포 핸드오프에서 수행한다.
- 스크린샷 QA는 필요한 범위만 수행한다.
- 변경 없는 영역은 "변경 없음"으로 기록한다.

============================================================
14. Loop Engineering 연결 정책
============================================================

계층형 개발계획서도 Loop Engineering 규칙을 따른다.

모든 개발 작업은 아래 루프를 따른다.

평가 -> 제안 -> 검증 -> 합치기 -> QA 평가 -> log 기록

절대 규칙:

1. 1사이클당 1가지 문제만 수정한다.
2. 실패한 루프는 기록한다.
3. 실패한 방식은 반복하지 않는다.
4. 검증 전에는 합치지 않는다.
5. QA 실패 시 완료 처리하지 않는다.
6. 3회 이상 루프 후 더 좋은 성공 루프는 Best Known Loop로 승격한다.
7. 단, 기존 성공 루프는 삭제하지 않는다.

계층형 계획과 Loop의 연결:

- Main Development Plan은 전체 루프 방향을 정한다.
- Step Detail Plan은 각 단계 루프의 실행 기준이 된다.
- Nested Detail Plan은 복잡한 하위 루프의 기준이 된다.
- Step Handoff는 루프 결과를 다음 단계로 넘긴다.
- Final Deployment Handoff는 전체 루프 결과를 종합한다.

18. 전역 AGENTS.md 추가 규칙
============================================================

## Hierarchical Development Planning Rules

1. For a new project, Codex must create a Main Development Plan before implementation.

2. The Main Development Plan must be based on sufficient requirement analysis, reasoning, prediction, risk analysis, scope definition, and evaluation criteria.

3. The Main Development Plan must not contain every implementation detail if doing so would make the document too long.

4. Each major phase in the Main Development Plan must be linked to a separate Step Detail Plan.

5. Step Detail Plans must be stored as separate files and connected through PLAN_INDEX.md.

6. If a Step Detail Plan becomes too complex, Codex may create Nested Detail Plans.

7. Codex must use the Main Development Plan as the overall control document.

8. During development, Codex must review only the active Step Detail Plan and necessary Nested Detail Plans to reduce token usage.

9. Codex must perform step-level handoff after each phase.

10. Codex must update PLAN_INDEX.md after each phase handoff.

11. Codex must not proceed to the next phase unless the current phase handoff is complete.

12. Scope expansion is prohibited unless the user approves it.

13. New ideas discovered during development must be recorded as Scope Change Candidates, not implemented immediately.

14. The purpose of hierarchical planning is to improve deliverable quality, reduce user intervention, prevent direction changes, prevent unnecessary feature expansion, and reduce token waste.

15. Final deployment handoff must summarize all completed steps, linked plans, deliverables, QA results, UI confidence, and remaining risks.

## UI Preview, Step QA, and Final Deployment Confidence Rules

1. If a project includes UI, GUI, dashboard, app screen, web page, widget, or user-facing visual output, Codex must create a UI Expected Preview before UI implementation.

2. The UI Expected Preview must include short annotations for each major UI area and function.

3. The user must approve the UI Expected Preview before Codex implements UI code.

4. The approved UI Expected Preview becomes part of the fixed evaluation criteria.

5. Step-level UI development must be evaluated against the approved UI preview for the specific UI part being implemented.

6. The required Part-level UI Match Confidence is 95% or higher.

7. Final deployment handoff must evaluate the full implemented UI against the approved full UI Expected Preview.

8. The required Final UI Match Confidence is 95% or higher.

9. Full UI Match Confidence is required at final deployment handoff, not at every intermediate step.

10. At intermediate steps, Codex must evaluate only the changed UI area to reduce unnecessary token usage.

11. If the execution environment allows it, Codex must run the actual program and use screenshots or visual inspection for UI QA.

12. If screenshot QA cannot be performed, Codex must record the reason and provide an alternate verification method.

13. UI scope expansion during implementation is prohibited unless separately approved by the user.

14. New UI ideas discovered during development must be recorded as future candidates, not silently implemented.

15. Codex must not repeat the full UI preview in every loop. It should reference the approved preview and summarize only the changed area.

16. The project main agent is responsible for final UI confidence verification during deployment handoff.

17. UI subagents must work only within the approved UI part assigned for the current loop.

21. PC Control Approval and Risk Operation Policy
============================================================

21.1 정책 목적

개발이 진행될 때 Codex가 PC 제어를 통해 작업해야 하는 경우가 있다.

예를 들어 다음과 같은 작업이 해당된다.

- 프로그램 실행
- GUI 조작
- 브라우저 실행
- 파일 탐색기 조작
- 설치 프로그램 실행
- 환경설정 변경
- 터미널 또는 PowerShell 명령 실행
- 테스트를 위한 앱 구동
- 스크린샷 확보
- 자동화 도구 실행
- 로컬 파일 생성, 수정, 삭제
- Git 명령 실행
- 배포 명령 실행

이때 Codex는 PC 제어가 필요한 작업을 임의로 생략하거나, 사용자가 모르는 상태에서 위험작업을 진행해서는 안 된다.

따라서 PC 제어가 필요한 경우에는 작업 성격에 따라 사용자 승인 기준을 구분한다.

핵심 목적은 다음과 같다.

1. PC 제어가 필요한 작업을 임의로 건너뛰지 않는다.
2. PC 제어가 필요하면 사용자에게 명확히 알린다.
3. 일반 PC 제어 작업은 사용자 승인 후 자체 진행한다.
4. 위험작업은 반드시 명시적 사용자 승인을 받은 뒤 진행한다.
5. 승인되지 않은 위험작업은 진행하지 않고 핸드오프에 기록한다.
6. PC 제어가 불가능한 경우 사유와 대체 검증 방법을 기록한다.


21.2 기존 전역 에이전트 정책과의 충돌 검토

이번 정책은 기존 전역 에이전트 정책과 충돌하지 않는다.

오히려 기존 정책을 보완한다.

| 기존 정책 | 충돌 여부 | 검토 결과 |
|---|---|---|
| Mandatory Pre-Implementation Gate | 충돌 없음 | PC 제어 전 승인 절차를 추가로 강화함 |
| 1 Loop 1 Change Rule | 충돌 없음 | PC 제어도 하나의 루프 목표 안에서만 수행 |
| Failed Loop Reuse 금지 | 충돌 없음 | PC 제어 실패도 실패 루프에 기록 가능 |
| QA 전 완료 금지 | 충돌 없음 | PC 제어 기반 검증을 QA 근거로 사용 |
| UI Screenshot QA | 충돌 없음 | 스크린샷 확보를 위한 PC 제어 승인 기준을 명확화 |
| 사용자 개입 최소화 | 부분 보완 | 위험작업은 사용자 개입이 필요하지만 안전을 위해 필수 |
| 토큰 절약 정책 | 충돌 없음 | 승인 기준을 표준화해 반복 설명을 줄임 |
| 단계별 핸드오프 | 충돌 없음 | PC 제어 수행 여부와 승인 여부를 핸드오프에 기록 |

판단:

PC Control Approval Policy는 기존 전역 에이전트와 충돌하지 않는다.
다만 "사용자 개입 최소화" 정책과는 균형 조정이 필요하다.

해결 기준:

- 일반 작업은 사전 승인 후 묶어서 자체 진행한다.
- 위험작업은 매번 명시적 승인을 받는다.
- 승인 기준을 미리 개발계획서에 정의해 불필요한 반복 질문을 줄인다.


21.3 PC 제어 작업 분류

PC 제어 작업은 위험도에 따라 3단계로 분류한다.

| 등급 | 구분 | 설명 | 승인 방식 |
|---|---|---|---|
| Level 1 | 일반 PC 제어 | 실행, 화면 확인, 스크린샷, 테스트 구동 | 사용자 1회 승인 후 해당 루프 내 자체 진행 |
| Level 2 | 주의 작업 | 파일 생성/수정, 설정 변경, 패키지 설치, Git commit | 작업 전 사용자 승인 필요 |
| Level 3 | 위험 작업 | 삭제, 덮어쓰기, 배포, 외부 전송, 인증정보 변경, 결제, 시스템 변경 | 반드시 명시적 사용자 승인 필요 |

21.4 Level 1 일반 PC 제어

Level 1은 비교적 안전한 작업이다.

예시:

- 프로그램 실행
- 테스트 목적의 앱 구동
- 로컬 웹 서버 실행
- 브라우저에서 localhost 확인
- 스크린샷 확보
- 화면 상태 확인
- 로그 확인
- 읽기 전용 파일 열람
- 빌드 또는 테스트 명령 실행
- UI QA를 위한 화면 캡처

승인 기준:

사용자가 해당 루프에서 PC 제어 진행을 승인하면 Codex는 해당 루프 범위 안에서 자체 진행할 수 있다.

단, 승인된 루프 범위를 벗어나면 다시 승인받아야 한다.

기록 기준:

핸드오프 문서에 다음 내용을 기록한다.

- PC 제어 수행 여부
- 실행한 프로그램 또는 명령
- 스크린샷 경로
- 로그 경로
- 검증 결과


21.5 Level 2 주의 작업

Level 2는 결과가 파일 또는 개발환경에 영향을 줄 수 있는 작업이다.

예시:

- 파일 생성
- 파일 수정
- 설정 파일 변경
- 의존성 설치
- 패키지 업데이트
- 가상환경 생성
- 테스트 데이터 생성
- Git add
- Git commit
- 브랜치 생성
- 로컬 설정 변경
- 자동화 스크립트 실행

승인 기준:

Codex는 작업 전에 다음 항목을 사용자에게 제시해야 한다.

1. 수행할 작업
2. 변경될 파일 또는 환경
3. 예상 영향
4. 되돌리는 방법
5. 검증 방법

사용자 승인 후 진행한다.

허용 승인 문구 예시:

승인.
이번 루프 범위 안에서 제안한 Level 2 작업만 진행해.

금지 사항:

- 승인 없이 파일 수정
- 승인 없이 설정 변경
- 승인 없이 패키지 설치
- 승인 없이 Git commit


21.6 Level 3 위험 작업

Level 3은 반드시 명시적 사용자 승인을 받아야 하는 위험작업이다.

예시:

- 파일 삭제
- 폴더 삭제
- 기존 파일 덮어쓰기
- 대량 파일 변경
- Git reset
- Git clean
- Git push
- Git merge
- Git rebase
- 배포 명령 실행
- 서버 설정 변경
- 운영 DB 접근
- 외부 서비스로 데이터 전송
- 이메일 또는 메시지 발송
- API Key, 토큰, 비밀번호 변경
- 결제 또는 비용 발생 작업
- 시스템 PATH 변경
- 보안 설정 변경
- 관리자 권한 명령 실행
- 자동 실행 서비스 등록
- 레지스트리 변경
- 원격 PC 제어 중 되돌리기 어려운 조작

승인 기준:

Level 3 작업은 반드시 개별 승인받아야 한다.

Codex는 아래 형식으로 사용자에게 확인해야 한다.

### 사용자 승인 필요
아래 문구로 승인해야 진행 가능:

제안한 작업만 진행해.

중요:

일반적인 "진행해", "승인"만으로는 Level 3 위험작업을 진행하지 않는다.
위험작업은 반드시 "위험작업 승인"과 같이 명시적으로 승인되어야 한다.


21.7 PC 제어를 임의로 건너뛰지 않는 규칙

Codex는 개발계획서 또는 세부계획서상 PC 제어가 필요한 작업을 임의로 건너뛰면 안 된다.

예시:

- UI QA에 스크린샷이 필요한데 실행하지 않음
- 프로그램 구동 검증이 필요한데 빌드만 하고 완료 처리
- 실제 실행 확인이 필요한데 코드만 보고 완료 처리
- 자동화 프로그램인데 실제 실행 테스트를 생략
- 사용자가 기대한 화면 검증 없이 완료 처리

처리 기준:

1. PC 제어가 가능하면 사용자 승인 후 진행한다.
2. PC 제어가 불가능하면 불가능한 사유를 기록한다.
3. 대체 검증 방법을 제안한다.
4. 핸드오프 문서에 미수행 사유를 남긴다.
5. 필요한 경우 해당 단계는 부분완료 또는 보류로 처리한다.

21.9 AGENTS.md 추가 규칙

전역 AGENTS.md와 프로젝트 AGENTS.md에 아래 규칙을 추가한다.

## PC Control and Dangerous Operation Approval Rules

1. Codex must not silently skip work that requires PC control.

2. If PC control is required, Codex must identify the required action and classify it as Level 1, Level 2, or Level 3.

3. Level 1 PC control may be performed after user approval within the approved loop scope.

4. Level 2 caution operations require user approval before execution.

5. Level 3 dangerous operations require explicit dangerous-operation approval.

6. A generic approval such as "승인" or "진행해" is not sufficient for Level 3 dangerous operations.

7. Level 3 operations require an explicit phrase such as "위험작업 승인".

8. Codex must explain the target, impact, risk, rollback method, and safer alternative before requesting approval for Level 3 operations.

9. If PC control cannot be performed, Codex must record the reason and propose an alternate verification method.

10. PC control execution results must be recorded in the step handoff document.

11. UI screenshot QA, program execution QA, and automation test execution must not be marked complete if required PC control was skipped without reason.

12. The project main agent is responsible for confirming that PC control approvals are properly recorded.

13. Subagents must not perform PC control outside their assigned loop scope.

14. Dangerous operations must never be bundled into a broad approval.


21.10 상태 판정에 추가되는 실패 조건

아래 조건 중 하나라도 해당하면 해당 루프 또는 단계는 실패 또는 보류로 판단한다.

- PC 제어가 필요한데 임의로 생략함
- PC 제어 생략 사유를 기록하지 않음
- Level 2 작업을 승인 없이 수행함
- Level 3 위험작업을 명시적 승인 없이 수행함
- 위험작업의 영향, 되돌리는 방법, 대체 방법을 설명하지 않음
- 승인된 범위를 넘어 PC 제어를 수행함
- 실행 결과를 핸드오프에 기록하지 않음
- 스크린샷 QA가 필요한데 수행하지 않고 완료 처리함


21.11 최종 원칙

PC 제어가 필요한 작업은 임의로 넘어가지 않는다.
일반 PC 제어는 승인 후 자체 진행한다.
주의 작업은 사전 승인 후 진행한다.
위험작업은 반드시 명시적 위험작업 승인을 받은 뒤 진행한다.
승인되지 않은 위험작업은 하지 않는다.
수행하지 못한 작업은 사유와 대체 검증 방법을 기록한다.
핸드오프 문서에 PC 제어 수행 결과를 남긴다.
