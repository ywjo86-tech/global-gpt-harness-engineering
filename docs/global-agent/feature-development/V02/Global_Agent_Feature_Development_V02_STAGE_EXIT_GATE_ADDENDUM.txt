2. 추가 보완 목적
============================================================

단계별 개발이 진행될 때, Codex가 각 단계를 단순히 완료 처리하고 다음 단계로 넘어가면 다음 문제가 발생할 수 있다.

1. 알려진 문제가 남은 상태로 다음 단계 진행
2. 테스트 실패를 무시하고 다음 단계 진행
3. 보안 위험을 기록하지 않음
4. 다음 단계에서 이전 단계 문제가 누적됨
5. 해결 가능한 문제를 미루어 재작업 비용 증가
6. 해결 불가능한 문제와 보류 항목이 명확히 구분되지 않음
7. 사용자가 실제 완료 상태를 오해함

이를 방지하기 위해 각 단계 종료 시 Stage Exit Gate를 적용한다.

Stage Exit Gate의 목적:

- 각 단계가 끝날 때 자체 점검을 수행한다.
- 알려진 문제, 테스트 실패, 보안 위험, 다음 단계 권장안을 확인한다.
- 같은 단계 안에서 해결 가능한 문제는 먼저 보완한다.
- 계획서 범위를 벗어나는 기능 확장이나 위험 작업은 임의로 구현하지 않는다.
- 현재 단계에서 해결할 수 없는 문제는 Known Issues와 Deferred Items로 기록한다.
- 다음 단계로 넘어가기 전 이전 단계의 완료 기준 충족 여부를 확인한다.


============================================================
3. Stage-by-Stage Progress Principle
============================================================

아래 원칙을 전역 에이전트와 프로젝트 에이전트에 추가한다.

## Stage-by-Stage Progress Principle

1. At the end of each stage, Codex must self-check known issues, test failures, security risks, and recommended next steps.

2. Issues that can be resolved within the current stage scope must be fixed before moving to the next stage.

3. Codex must not implement out-of-scope feature expansion or dangerous operations without user approval.

4. Issues that cannot be resolved in the current stage must be recorded in the handoff document as Known Issues and Deferred Items.

5. Before moving to the next stage, Codex must verify that the completion criteria of the previous stage have been satisfied.

6. If the previous stage completion criteria are not satisfied, Codex must not mark the stage as complete.

7. If an issue is within the current stage scope and safe to fix, Codex may propose a same-stage remediation loop.

8. A same-stage remediation loop must still follow the one-loop-one-change rule.

9. If the fix requires scope expansion, Level 2 caution work, or Level 3 dangerous work, Codex must request user approval before proceeding.

10. The project main agent is responsible for deciding whether the stage can proceed, must be remediated, or must be deferred.


============================================================
4. Stage Exit Gate
============================================================

각 단계가 끝날 때 Codex는 아래 Stage Exit Gate를 통과해야 한다.

## Stage Exit Gate Checklist

| No | Check Item | Required Result |
|---|---|---|
| 1 | Stage objective completed | Yes |
| 2 | Completion criteria satisfied | Yes |
| 3 | Known issues reviewed | Yes |
| 4 | Test failures reviewed | Yes |
| 5 | Security risks reviewed | Yes |
| 6 | Solvable issues fixed within current stage | Yes or Not Applicable |
| 7 | Out-of-scope items not implemented | Yes |
| 8 | Dangerous operations not performed without approval | Yes |
| 9 | Deferred items recorded | Yes or Not Applicable |
| 10 | Next stage recommendation written | Yes |
| 11 | Handoff document updated | Yes |
| 12 | Progress percentage updated | Yes |

Stage Exit Result:

- GO: 다음 단계 진행 가능
- CONDITIONAL GO: 기록된 Deferred Items가 있으나 다음 단계 진행 가능
- NO-GO: 완료 기준 미충족으로 다음 단계 진행 금지


============================================================
5. Same-Stage Remediation Rule
============================================================

같은 단계 안에서 해결 가능한 문제는 다음 단계로 넘기지 않고 먼저 보완한다.

다만 다음 조건을 모두 만족해야 한다.

1. 현재 단계 범위 안의 문제일 것
2. 기존 승인된 개발계획서와 충돌하지 않을 것
3. 위험작업이 아닐 것
4. 새로운 기능 확장이 아닐 것
5. 1사이클 1수정 원칙으로 처리 가능할 것
6. 검증 방법이 명확할 것

예시:

허용:
- 현재 단계에서 작성한 템플릿의 누락 항목 보완
- 현재 단계 테스트 실패 1건 수정
- 현재 단계 문서 내 링크 오류 수정
- 현재 단계 QA 체크리스트 누락 보완
- 현재 단계 UI 영역의 부분 신뢰도 미달 항목 수정

금지:
- 다음 단계 기능을 미리 구현
- 계획서에 없는 신규 기능 추가
- 위험작업을 승인 없이 진행
- 파일 삭제 또는 대량 변경
- 전체 구조 리팩토링
- 여러 테스트 실패를 한 루프에서 동시에 수정


============================================================
6. Known Issues와 Deferred Items 구분
============================================================

현재 단계에서 발견된 문제는 아래 두 가지로 구분한다.

## Known Issues

현재 단계 또는 전체 프로젝트에 영향을 줄 수 있는 알려진 문제이다.

예시:

- 테스트 1건 실패
- UI 예상도와 부분 불일치
- 보안 검토 필요
- 특정 환경에서 실행 확인 불가
- 의존성 버전 확인 필요

## Deferred Items

현재 단계에서는 처리하지 않고 다음 단계 또는 후속 버전으로 넘기는 항목이다.

예시:

- 계획서 범위를 벗어나는 개선 아이디어
- 다음 단계에서 처리하기로 예정된 기능
- 사용자 승인 필요 항목
- 위험작업 승인 대기 항목
- 현재 환경에서 검증 불가능한 항목
- 최종 배포 단계에서 종합 검수할 항목

8. AGENTS.md 추가 규칙
============================================================

아래 내용을 전역 AGENTS.md와 프로젝트 AGENTS.md에 추가한다.

## Stage Exit Gate and Deferred Issue Rules

1. At the end of each stage, Codex must perform a Stage Exit Review.

2. The Stage Exit Review must check known issues, test failures, security risks, completion criteria, and recommended next steps.

3. Issues that can be fixed within the current stage scope must be resolved before moving to the next stage.

4. Same-stage fixes must follow the one-loop-one-change rule.

5. Codex must not implement out-of-scope features during same-stage remediation.

6. Codex must not perform Level 2 caution operations or Level 3 dangerous operations without the required user approval.

7. Issues that cannot be resolved in the current stage must be recorded as Known Issues or Deferred Items.

8. Before moving to the next stage, Codex must confirm that the previous stage completion criteria are satisfied.

9. If the completion criteria are not satisfied, Codex must mark the stage as NO-GO and must not proceed.

10. If unresolved items do not block the next stage, Codex may mark the stage as CONDITIONAL GO and record Deferred Items.

11. The project main agent is responsible for the final Stage Exit Decision.

12. The handoff document must include Stage Exit Review, Known Issues, Deferred Items, Test Failures, Security Risks, and Next Stage Recommendation.

9. 기존 V02 정책과 충돌 검토
============================================================

이번 추가 보완은 기존 V02 정책과 충돌하지 않는다.

| 기존 정책 | 충돌 여부 | 검토 결과 |
|---|---|---|
| 1 Loop 1 Change Rule | 충돌 없음 | Same-stage fix도 1루프 1수정 원칙을 따름 |
| Hierarchical Planning | 충돌 없음 | 단계별 세부계획서 종료 게이트를 강화함 |
| Step Handoff | 충돌 없음 | Handoff에 Known Issues와 Deferred Items를 추가함 |
| UI QA 95% | 충돌 없음 | 미달 항목은 같은 단계에서 보완하거나 Deferred로 기록 |
| PC Control Approval | 충돌 없음 | 위험작업은 기존 승인 정책을 그대로 따름 |
| Scope Control | 충돌 없음 | 범위 확장 금지 원칙을 더 명확히 함 |
| Token Efficiency | 충돌 없음 | 해결 가능한 문제만 같은 단계에서 보완하고, 나머지는 기록하여 반복 분석을 줄임 |
| User Intervention Minimization | 충돌 없음 | 승인 필요 지점만 명확히 하고 불필요한 질문을 줄임 |

최종 판단:

Stage Exit Gate는 기존 전역 에이전트 V02 정책과 충돌하지 않는다.
오히려 단계 종료 품질을 높이고, 다음 단계로 문제를 누적시키지 않으며, 해결 불가능한 문제를 명확히 기록하는 보완 규칙이다.

Required new policy:

1. At the end of each stage, Codex must self-check known issues, test failures, security risks, and next-stage recommendations.
2. Issues that can be resolved within the current stage scope must be fixed before moving to the next stage.

Before modifying files, perform the mandatory pre-implementation gate:

1. Evaluate the current repository structure.
2. Identify where the existing V02 policy was added.
3. Propose files to modify.
4. Propose the first loop target.
5. Request user approval.

Important:
- One loop may modify only one target.
- Do not modify unrelated files.
- Do not perform dangerous operations without explicit approval.
- If this requires PC control, classify it as Level 1, Level 2, or Level 3 first.
- After the change, update the handoff with progress, known issues, deferred items, and next-stage recommendation.

Wait for user approval before making changes.

12. 최종 원칙
============================================================

각 단계는 그냥 끝내지 않는다.
끝날 때마다 자체 점검한다.
해결 가능한 문제는 같은 단계에서 보완한다.
범위 밖 기능 확장은 하지 않는다.
위험작업은 반드시 승인받는다.
해결할 수 없는 문제는 Known Issues와 Deferred Items로 기록한다.
다음 단계 전에는 이전 단계 완료 기준을 확인한다.
완료 기준을 만족하지 못하면 다음 단계로 넘어가지 않는다.
