# Request Intent Definition Harness Pattern

## Purpose

Use this pattern to turn a broad or ambiguous user request into a clear purpose, requirements, constraints, assumptions, and final deliverable definition.

## When to Use

- The user request is underspecified.
- The requested action may not match the actual business goal.
- Requirements, constraints, or output format are unclear.
- Implementation would be risky without clarification.

## Agent Team Composition

- Intent lead: identifies the real purpose behind the surface request.
- Requirements analyst: defines functional and non-functional needs.
- Constraint analyst: identifies boundaries, risks, and non-goals.
- Ambiguity reviewer: separates blockers from safe assumptions.
- Deliverable framer: defines the final output and acceptance criteria.

## Workflow

1. Capture the surface request exactly.
2. Infer the likely real purpose and decision need.
3. Identify requirements and expected outcomes.
4. Capture constraints, risks, non-goals, and sensitive data concerns.
5. List ambiguities and classify them by risk.
6. Define final deliverable format and acceptance criteria.
7. Recommend the next harness or execution path.

## Inputs

- User request
- Available context
- Stakeholder or audience
- Known constraints
- Desired decision or output

## Outputs

- Surface request summary
- Real purpose statement
- Requirements
- Constraints and non-goals
- Ambiguity list
- Final deliverable definition
- Recommended next action

## Validation Criteria

- The purpose explains why the work matters.
- Requirements are clear enough to execute.
- Blockers and assumptions are separated.
- Final output is concrete and testable.

## Cautions

- Do not over-ask when safe assumptions are possible.
- Do not proceed to implementation if a high-risk ambiguity remains.
- Do not bury sensitive data concerns in general notes.
