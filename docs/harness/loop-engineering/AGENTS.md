# Codex Loop Engineering 규칙

## 위치

Loop Engineering은 이 저장소의 코드 작업 운영 규칙이다.
전역 스캐폴드는 `.codex-loop-global/`에 있고, 루트 에이전트가 정책을 보유하며, 프로젝트 에이전트가 이를 적용하고, 서브에이전트가 그 안에서 실행한다.
서브에이전트는 원본 정책을 수정할 수 없다.

## 절대 규칙

1. 모든 코드 작업은 `평가 -> 제안 -> 검증 -> 합치기 -> QA 평가 -> 기록` 순서를 따른다.
2. 하나의 루프는 하나의 문제 또는 하나의 대상만 수정한다.
3. 구현을 시작하기 전에 평가 기준을 먼저 고정한다.
4. 평가 기준이 없으면 정량 기준을 제안하고, 코드 작성 전에 승인을 요청한다.
5. 승인된 기준은 해당 작업이 끝날 때까지 고정한다.
6. 실패한 루프는 반드시 `.codex-loop/logs/failed-loop-log.md`에 기록한다.
7. 새 루프를 시작하기 전에 실패 기록을 먼저 확인한다.
8. 실패한 접근 방식을 다시 사용하지 않는다.
9. 검증이 끝나기 전에는 합치지 않는다.
10. 합친 뒤에는 QA를 수행한다.
11. 성공이든 실패든 모든 루프를 기록한다.
12. 최소 3회 이상 루프가 진행된 뒤에만 성공 루프를 정량 비교한다.
13. 기존 성공 기록은 삭제하지 말고 `Superseded`로 남긴다.
14. 가장 검증된 방식은 `.codex-loop/memory/best-known-loops.md`에 둔다.
15. 서브에이전트는 자기 역할 범위 안에서만 작업한다.
16. 프로젝트 메인 에이전트가 루프 상태, 합치기 적합성, QA 결과, 기록 관리를 최종 책임진다.

## Global Agent V02 Planning Notes

- Global Agent 계열 작업에서는 `MAIN_DEVELOPMENT_PLAN.md`를 통제 문서로 보고, 현재 루프는 활성 `STEP_*_DETAIL_PLAN.md`만 참조한다.
- 계획이 복잡해지면 메인 계획을 키우지 말고 nested detail plan을 추가한다.
- 각 단계 완료 시 `HANDOFF_*` 문서로 결과와 다음 단계 인계를 기록한다.
- 새 아이디어는 즉시 범위에 넣지 말고 scope candidate로 기록한다.
- UI 작업이 있으면 먼저 예상 화면을 만들고, 단계 수준에서는 변경된 UI 부분만 검증한다.
- PC control은 Level 1, Level 2, Level 3로 분류하며, Level 2는 사전 승인, Level 3는 명시적 위험작업 승인을 요구한다.
- PC control 실행 결과와 미수행 사유는 관련 handoff 파일에 기록한다.

## 루프 절차

### 1. 평가
- 요구사항을 이해한다.
- 정확한 대상을 식별한다.
- 테스트, 로그, 재시도 이력을 확인한다.
- 고정 평가 기준을 확인한다.

### 2. 제안
- 한 번에 하나만 제안한다.
- 대상 파일을 명시한다.
- 예상 영향 범위를 적는다.
- 역할 범위를 벗어나지 않는지 확인한다.

### 3. 검증
- 최소한으로 의미 있는 검증을 수행한다.
- 검증이 실패하면 합치지 않는다.
- 필요하면 실패 원인을 기록한다.

### 4. 합치기
- 검증된 변경만 합친다.
- 무관한 수정은 포함하지 않는다.

### 5. QA 평가
- 고정 기준과 비교해 결과를 확인한다.
- 회귀와 역할 위반이 없는지 확인한다.

### 6. 기록
- 성공 또는 실패를 기록한다.
- 필요하면 재사용 금지 패턴과 Best Known 업데이트를 추가한다.

## 금지 사항

- 한 루프에 여러 문제를 섞지 않는다.
- 실패한 수정을 반복하지 않는다.
- 평가 기준을 건너뛰지 않는다.
- 검증되지 않은 코드를 합치지 않는다.
- 평가 기준을 몰래 바꾸지 않는다.
- 이전 성공 기록을 삭제하지 않는다.
- 서브에이전트가 원본 정책을 바꾸게 두지 않는다.
