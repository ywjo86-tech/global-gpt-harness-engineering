# Codex Loop Engineering DEVELOPMENT PLAN
# 최종본
# 파일명: Loop_Engineering_DEVELOPMENT_PLAN.txt

문서 버전: v1.0 Final
작성 목적: Codex 코드작업 전용 Loop Engineering 개발 및 적용 계획 수립
핵심 포지션: Loop Engineering은 전역 운영체계이며, 메인 에이전트는 루프 관리자이고, 서브에이전트는 루프 실행자이다.

============================================================
1. 문서 목적
============================================================

본 문서는 Codex 코드작업을 안정적으로 운영하기 위한 Codex Loop Engineering의 최종 개발 계획서이다.

Codex Loop Engineering은 특정 프로젝트, 특정 업무, 특정 기능에 종속되는 개발 방식이 아니다.
이는 Codex가 모든 코드작업을 수행할 때 따라야 하는 전역 운영 규칙이다.

Codex Loop Engineering은 다음 문제를 방지하기 위해 설계한다.

1. 요구사항을 충분히 평가하지 않은 상태에서 코드 수정
2. 한 번에 여러 문제를 수정하여 코드가 꼬이는 문제
3. 실패한 해결책을 반복 적용하는 무한루프
4. 검증되지 않은 코드를 병합하는 문제
5. 성공·실패 루프가 기록되지 않아 개선 이력이 남지 않는 문제
6. 기존 성공 방식보다 더 좋은 방식이 발견되어도 반영되지 않는 문제
7. 전역 에이전트, 프로젝트 에이전트, 서브에이전트 간 역할이 불명확해지는 문제

Codex는 모든 코드작업에서 아래 루프를 1사이클로 수행해야 한다.

평가 -> 제안 -> 검증 -> 합치기 -> QA 평가 -> log 기록


============================================================
2. Loop Engineering의 최종 포지셔닝
============================================================

Loop Engineering은 하나의 서브에이전트가 아니다.
Loop Engineering은 Codex 코드작업 전체를 통제하는 전역 운영체계이다.

최종 포지션은 다음과 같다.

전역 글로벌 에이전트
  └─ Loop Engineering 원본 규칙 보유
      └─ 각 프로젝트 AGENTS.md에 상속
          └─ 프로젝트별 서브에이전트에 실행 규칙으로 반영

정리하면 다음과 같다.

원본 규칙 = 전역 메인 에이전트
실행 적용 = 프로젝트 AGENTS.md
세부 수행 = 서브에이전트

Loop Engineering은 전역 운영체계이고,
메인 에이전트는 루프 관리자이며,
서브에이전트는 루프 안에서 움직이는 실행자다.


============================================================
3. 에이전트 계층 구조
============================================================

권장 구조는 다음과 같다.

Global Agent
  └─ Loop Engineering Policy
      ├─ Project Main Agent
      │   ├─ Frontend Subagent
      │   ├─ Backend Subagent
      │   ├─ QA Subagent
      │   ├─ Logger Subagent
      │   ├─ Test Subagent
      │   └─ Refactor Subagent
      └─ .codex-loop Memory

각 계층의 역할은 다음과 같다.

1. 전역 글로벌 에이전트
- Loop Engineering 원본 규칙 보유
- 전체 Codex 코드작업의 운영 원칙 정의
- 평가 기준 수립 방식 정의
- 실패 루프 재사용 금지 정책 정의
- 성공 루프 개선 정책 정의
- 서브에이전트가 변경할 수 없는 최상위 규칙 관리

2. 프로젝트 메인 에이전트
- 전역 Loop Engineering 규칙을 프로젝트에 상속
- 프로젝트별 AGENTS.md 관리
- 프로젝트별 평가 기준 관리
- 루프 실행 상태 관리
- 성공·실패·개선 루프 최종 기록 관리
- Best Known Loop 승격 여부 판단
- 서브에이전트 작업 결과를 검토하고 합치기 여부 판단

3. 서브에이전트
- 특정 작업을 수행하는 실행자
- 전역 Loop Engineering 규칙을 반드시 따름
- 자기 역할 범위 안에서만 작업
- 1사이클 1수정 원칙 준수
- 실패 시 로그 기록에 필요한 정보를 제공
- 검증 실패 시 병합 요청 금지
- Loop Engineering 원본 규칙을 임의 변경할 수 없음

4. QA 서브에이전트
- 검증 및 QA 평가 담당
- 고정 평가 기준 충족 여부 확인
- 신규 오류 및 회귀 오류 확인
- QA 실패 시 완료 처리 금지

5. Logger 서브에이전트
- 성공 루프 기록
- 실패 루프 기록
- 개선 성공 루프 기록
- 재사용 금지 실패 방식 기록
- Best Known Loop 기록 보조

중요 원칙:

Loop Engineering은 전역에 존재한다.
프로젝트는 Loop Engineering을 상속한다.
서브에이전트는 Loop Engineering을 따른다.
서브에이전트는 Loop Engineering을 임의 변경할 수 없다.
성공/실패/개선 루프의 최종 판단은 프로젝트 메인 에이전트가 관리한다.

============================================================

Codex Loop Engineering의 최종 목표는 다음과 같다.

작게 수정한다.
먼저 평가한다.
기준을 고정한다.
하나만 바꾼다.
검증 전에는 합치지 않는다.
실패는 기록한다.
실패한 방식은 반복하지 않는다.
성공도 비교한다.
더 좋은 성공 루프는 승격한다.
기존 성공 이력은 삭제하지 않는다.
서브에이전트는 실행만 하고 원본 규칙은 수정하지 않는다.
QA까지 통과해야 완료다.

============================================================
6. 핵심 개념
============================================================

6.1 Loop

Loop는 Codex가 코드작업을 수행하는 최소 단위이다.

하나의 Loop는 반드시 아래 순서로 진행한다.

1. 평가
2. 제안
3. 검증
4. 합치기
5. QA 평가
6. log 기록


6.2 Cycle

Cycle은 하나의 Loop 실행 단위이다.

1 Cycle = 1개의 문제 또는 1개의 수정 목표

예시:

Cycle 1: 로그인 버튼 클릭 오류 수정
Cycle 2: 테스트 실패 1건 수정
Cycle 3: UI 링 두께 조정


6.3 Fixed Evaluation Criteria

Fixed Evaluation Criteria는 작업 시작 전에 고정되는 정량 평가 기준이다.

Codex는 이 기준을 만족해야만 작업을 완료 처리할 수 있다.


6.4 Failed Loop

Failed Loop는 검증 또는 QA 평가에서 실패한 루프이다.

Failed Loop는 반드시 기록해야 하며, 다음 루프에서 동일한 접근 방식을 재사용하면 안 된다.


6.5 Success Loop

Success Loop는 평가 기준과 QA 기준을 모두 통과한 루프이다.

성공 루프도 기록해야 하며, 이후 더 나은 방식이 발견되면 기존 성공 루프를 삭제하지 않고 개선 루프로 승격한다.


6.6 Updated Success Loop

Updated Success Loop는 기존 성공 루프보다 더 좋은 방식으로 확인된 개선 성공 루프이다.


6.7 Best Known Loop

Best Known Loop는 3회 이상 루프가 진행된 뒤, 기존 성공 루프보다 더 좋은 방식으로 확인된 최적 성공 루프이다.


6.8 Global Policy

Global Policy는 전역 글로벌 에이전트에 위치한 Loop Engineering 원본 규칙이다.
서브에이전트는 Global Policy를 임의 변경할 수 없다.


6.9 Project Execution Rule

Project Execution Rule은 전역 Loop Engineering을 각 프로젝트에 적용한 프로젝트별 실행 규칙이다.


6.10 Subagent Execution Rule

Subagent Execution Rule은 서브에이전트가 자기 역할 범위 안에서 Loop Engineering을 따르기 위한 실행 규칙이다.


============================================================
7. 절대 규칙
============================================================

아래 규칙은 Codex Loop Engineering의 최상위 규칙이다.

Rule 1. 모든 코드작업은 Loop를 따른다

Codex는 모든 코드작업에서 아래 순서를 따라야 한다.

평가 -> 제안 -> 검증 -> 합치기 -> QA 평가 -> log 기록


Rule 2. 1사이클당 1가지 문제만 수정한다

하나의 Cycle에서는 반드시 하나의 수정 목표만 다룬다.

허용 예시:

- 로그인 버튼 클릭 오류 수정
- 특정 함수의 예외 처리 보완
- 테스트 실패 1건 수정
- UI 외곽 링 애니메이션 조정

금지 예시:

- 로그인 오류 수정 + UI 개선 + 폴더 구조 변경
- 테스트 수정 + 리팩토링 + 의존성 업데이트
- 성능 개선 + 기능 추가 + 디자인 변경


Rule 3. 평가 기준은 최초 설계단계에서 수립한다

Codex는 프로젝트 또는 기능 작업을 시작하기 전에 반드시 평가 기준을 수립해야 한다.

평가 기준은 정량적이어야 한다.

예시:

| 항목 | 기준 |
|---|---|
| 빌드 성공 여부 | 성공 |
| 테스트 통과율 | 100% |
| 신규 오류 수 | 0건 |
| 기존 기능 회귀 오류 | 0건 |
| 린트 오류 | 0건 |
| 타입 오류 | 0건 |
| 요구사항 충족률 | 100% |
| QA 체크리스트 충족률 | 100% |
| 1루프 1수정 원칙 | 준수 |
| 로그 기록 여부 | 필수 |


Rule 4. 평가 기준이 없으면 Codex가 추천한다

사용자가 별도의 평가 기준을 제공하지 않은 경우 Codex는 요구사항을 분석하여 평가 기준을 추천해야 한다.

Codex는 바로 코드 수정에 들어가면 안 된다.

진행 순서:

1. 요구사항 분석
2. 평가 기준 추천
3. 사용자 승인 요청
4. 승인된 기준 고정
5. 고정된 기준에 따라 Loop 실행


Rule 5. 승인된 평가 기준은 임의 변경 금지

사용자가 승인한 평가 기준은 해당 작업이 종료될 때까지 고정한다.

평가 기준 변경이 필요한 경우에는 별도 Loop로 처리한다.


Rule 6. 실패한 루프는 반드시 기록한다

실패한 루프는 반드시 failed-loop-log.md에 기록한다.

기록 항목:

- Loop ID
- 수정 목표
- 적용한 변경
- 검증 결과
- 실패 원인
- 재사용 금지 접근 방식
- 다음 루프에서 피해야 할 방식


Rule 7. 실패한 루프는 반복 재사용하지 않는다

Codex는 다음 루프를 시작하기 전에 반드시 실패 로그를 확인해야 한다.

동일한 실패 접근 방식은 재사용하면 안 된다.

금지 조건:

같은 파일 + 같은 원인 + 같은 수정 방식 = 재사용 금지


Rule 8. 검증 전에는 합치기 금지

검증 단계가 실패하면 합치기 단계로 이동할 수 없다.

검증 실패 시:

- 합치기 금지
- 실패 루프 기록
- 동일 접근 방식 재사용 금지
- 다음 루프에서 다른 방식 제안


Rule 9. QA 평가 실패 시 완료 처리 금지

합치기 후 QA 평가에서 실패하면 해당 루프는 실패로 기록한다.

QA 평가 실패 시 다음 루프에서는 같은 수정 방식을 반복하지 않는다.


Rule 10. 성공 루프도 기록한다

성공한 루프도 반드시 success-loop-log.md에 기록한다.

성공 로그는 이후 개선 비교의 기준이 된다.


Rule 11. 3회 이상 루프 후 성공 루프를 비교한다

전체 Cycle이 3회 이상 진행된 이후에는 성공 루프 간 비교가 가능하다.

기존 성공 루프보다 더 좋은 방식이 발견되면 해당 루프를 개선 성공 루프로 승격한다.


Rule 12. 기존 성공 루프는 삭제하지 않는다

더 좋은 성공 루프가 발견되어도 기존 성공 루프는 삭제하지 않는다.

처리 방식:

기존 성공 루프: 보존
기존 루프 상태: Superseded 처리
개선 루프: Updated Success Loop로 기록
최신 권장 루프: Best Known Loop로 승격


Rule 13. 성공 루프 개선은 정량 기준으로 판단한다

성공 루프 개선은 감각적 판단이 아니라 정량 기준으로 판단한다.

예시:

| 비교 항목 | 개선 판단 기준 |
|---|---|
| 수정 파일 수 | 기존보다 적을 것 |
| 코드 변경량 | 기존보다 적을 것 |
| 테스트 통과율 | 동일하거나 높을 것 |
| 신규 오류 수 | 0건 유지 |
| 회귀 오류 수 | 0건 유지 |
| 빌드 결과 | 성공 유지 |
| QA 결과 | 통과 필수 |
| 영향 범위 | 기존보다 좁을 것 |
| 유지보수성 | 기존보다 좋을 것 |


Rule 14. Loop Engineering 원본 규칙은 전역 메인 에이전트가 보유한다

Loop Engineering 원본 규칙은 전역 글로벌 에이전트에 위치한다.
프로젝트와 서브에이전트는 이를 상속하여 실행한다.


Rule 15. 서브에이전트는 Loop Engineering을 임의 변경할 수 없다

서브에이전트는 Loop Engineering을 실행할 수 있지만 원본 규칙을 변경할 수 없다.
변경이 필요한 경우 프로젝트 메인 에이전트가 검토하고, 전역 규칙 변경은 별도 승인 절차를 거친다.


Rule 16. 프로젝트 메인 에이전트가 루프 최종 판단권을 가진다

성공, 실패, 개선 성공, Best Known Loop 승격 여부는 프로젝트 메인 에이전트가 최종 관리한다.


Rule 17. 서브에이전트는 역할 범위를 초과할 수 없다

Frontend Subagent는 UI 범위 안에서만 작업한다.
Backend Subagent는 API 또는 서버 범위 안에서만 작업한다.
QA Subagent는 검증과 QA 평가를 담당한다.
Logger Subagent는 로그 기록을 담당한다.

# Codex Loop Engineering Rules

## Positioning

Loop Engineering is the global operating system for Codex code work.
The main agent manages the loop.
Subagents execute work inside the loop.
Subagents must not modify the original Loop Engineering policy.

## Absolute Rules

1. Every code task must follow the loop:
   Evaluate -> Propose -> Verify -> Merge -> QA Evaluate -> Log.

2. One loop cycle may modify only one problem or one target.

3. Evaluation criteria must be defined before coding.

4. If the user does not provide evaluation criteria, Codex must recommend quantitative criteria and ask the user to approve them.

5. Once approved, evaluation criteria are fixed for the task.

6. Failed loops must be recorded in:
   .codex-loop/logs/failed-loop-log.md

7. Codex must check failed-loop-log.md before starting a new loop.

8. Codex must not reuse a failed loop approach.

9. Code must not be merged before verification passes.

10. QA evaluation must be performed after merge.

11. Every loop must be logged, whether successful or failed.

12. After at least 3 loop cycles, Codex may compare successful loops.

13. If a later successful loop is proven to be better than a previous successful loop using quantitative criteria, Codex must not delete the old success loop.

14. The previous success loop must be marked as Superseded.

15. The improved loop must be recorded as an Updated Success Loop.

16. The best verified approach must be recorded in:
    .codex-loop/memory/best-known-loops.md

17. Successful loops may only be updated when the new loop passes the same fixed evaluation criteria and shows measurable improvement.

18. The original Loop Engineering policy belongs to the global main agent.

19. Project agents inherit the global Loop Engineering policy.

20. Subagents execute the inherited Loop Engineering rules but must not change the original policy.

21. Subagents must stay within their assigned role boundary.

22. The project main agent has final responsibility for loop status, merge decision, QA result, and log management.

## Loop Process

### 1. Evaluate
- Understand the user requirement.
- Identify the exact code target.
- Check existing tests, logs, and failed loop history.
- Confirm fixed evaluation criteria.
- Confirm responsible agent or subagent.

### 2. Propose
- Propose one change only.
- Identify target files.
- Explain expected impact.
- Define verification method.
- Confirm the proposed change is inside the assigned agent role.

### 3. Verify
- Run build, tests, lint, type checks, or equivalent validation.
- If verification fails, do not merge.
- Record failed loop.

### 4. Merge
- Merge only verified changes.
- Do not include unrelated changes.
- Project main agent must confirm merge suitability.

### 5. QA Evaluate
- Confirm the result against fixed evaluation criteria.
- Confirm no regression.
- Confirm no new error.
- Confirm subagent role boundary was not violated.

### 6. Log
- Record success or failure.
- Failed approaches must be added to forbidden retry patterns.
- Improved success loops must be recorded when applicable.
- Best Known Loop must be updated when applicable.

## Forbidden Behavior

- Do not modify multiple issues in one loop.
- Do not repeat a failed fix.
- Do not skip evaluation criteria.
- Do not merge unverified code.
- Do not perform broad refactoring during a bug-fix loop.
- Do not silently change the evaluation criteria.
- Do not delete previous success loops.
- Do not promote a better loop without quantitative evidence.
- Do not allow subagents to change the original Loop Engineering policy.
- Do not allow subagents to work outside their assigned role.
- Do not treat QA failure as completion.

완료 기준:
- 전역 AGENTS.md에 Loop Engineering 원본 규칙 반영
- 프로젝트 AGENTS.md에 상속 규칙 반영
- 서브에이전트 권한 제한 반영
- 프로젝트 메인 에이전트의 최종 판단 책임 반영

12. Codex 작업 요청 프롬프트 표준
============================================================

Codex에게 작업을 요청할 때 아래 프롬프트를 사용한다.

이번 작업은 Codex Loop Engineering 규칙을 반드시 적용해 진행해줘.

Loop Engineering 포지션:
1. Loop Engineering은 전역 운영체계다.
2. 메인 에이전트는 루프 관리자다.
3. 서브에이전트는 루프 실행자다.
4. 서브에이전트는 Loop Engineering 원본 규칙을 변경할 수 없다.
5. 프로젝트는 전역 Loop Engineering을 상속해서 실행한다.

절대 규칙:
1. 루프는 평가 -> 제안 -> 검증 -> 합치기 -> QA 평가 -> log 기록 순서로 진행한다.
2. 1사이클당 1가지 문제만 수정한다.
3. 평가 기준은 작업 전에 수립한다.
4. 별도 평가 기준이 없으면 Codex가 정량 평가 기준을 추천하고, 사용자 승인 후 고정한다.
5. 실패한 루프는 failed-loop-log.md에 기록한다.
6. 다음 루프에서는 실패한 접근 방식을 재사용하지 않는다.
7. 검증 실패 시 합치기 금지.
8. QA 평가 실패 시 완료 처리 금지.
9. 모든 루프는 성공/실패 여부와 관계없이 log를 남긴다.

14. 무한루프 방지 정책

.codex-loop/logs/failed-loop-log.md

무한루프 방지 조건:

예시:

실패 결과:
링 구조 자체가 목표 이미지와 달라 실패

재사용 금지:
단순 scale 조정 방식 반복 금지

다음 접근:
SVG 구조 또는 canvas ring layer 구조 자체를 재설계

15. 성공 루프 개선 정책
============================================================

성공 루프는 고정된 정답이 아니다.

3회 이상 루프가 진행된 후 기존 성공 루프보다 더 좋은 방식이 발견되면 성공 루프를 개선할 수 있다.

단, 기존 성공 루프는 삭제하지 않는다.

개선 처리 순서:

1. 전체 루프 3회 이상 진행 여부 확인
2. 기존 성공 루프 확인
3. 새로운 성공 루프와 정량 비교
4. 동일 평가 기준 충족 여부 확인
5. QA 통과 여부 확인
6. 프로젝트 메인 에이전트 승인
7. 기존 성공 루프 Superseded 처리
8. 개선 루프 Updated Success Loop로 기록
9. 개선 루프 Best Known Loop로 승격

정량 비교 기준:

| 항목 | 기준 |
|---|---|
| 수정 파일 수 | 기존보다 적거나 동일 |
| 코드 변경 라인 수 | 기존보다 적거나 명확히 단순 |
| 테스트 통과율 | 동일하거나 높음 |
| 신규 오류 수 | 0건 |
| 회귀 오류 수 | 0건 |
| QA 결과 | 통과 |
| 영향 범위 | 기존보다 좁음 |
| 유지보수성 | 기존보다 우수 |
| 재사용성 | 기존보다 우수 |

18. 기대 효과
============================================================

Codex Loop Engineering 적용 후 기대 효과는 다음과 같다.

- 코드 수정 범위가 작아진다.

- 전역 에이전트, 프로젝트 에이전트, 서브에이전트의 역할이 명확해진다.
- 서브에이전트가 독단적으로 규칙을 변경하는 문제가 방지된다.

============================================================
19. 최종 요약
============================================================

Codex Loop Engineering은 Codex가 코드를 무작정 수정하지 않고, 평가 기준을 먼저 수립한 뒤, 하나의 루프에서 하나의 문제만 수정하고, 검증과 QA를 통과한 경우에만 합치며, 모든 성공과 실패를 기록하는 작업 운영 체계이다.

또한 3회 이상 루프가 진행된 후 더 나은 성공 방식이 발견되면 기존 성공 루프를 삭제하지 않고 Superseded 처리하며, 개선된 방식을 Best Known Loop로 승격한다.

최종 포지셔닝은 다음과 같다.

Loop Engineering = 전역 운영체계
전역 메인 에이전트 = 원본 규칙 보유자
프로젝트 메인 에이전트 = 루프 관리자
서브에이전트 = 루프 실행자
QA 서브에이전트 = 검증 담당자
Logger 서브에이전트 = 기록 담당자

최종 운영 원칙은 다음과 같다.

평가한다.
제안한다.
검증한다.
합친다.
QA 평가한다.
기록한다.
실패는 반복하지 않는다.
성공은 개선한다.
한 번에 하나만 수정한다.
서브에이전트는 실행만 한다.
메인 에이전트가 루프를 관리한다.
