# Requirements Analysis

Use this skill after project intake and before solution design or implementation planning.

## Required Inputs

- Intake summary
- User goals
- Known constraints
- Existing docs or examples, if available

## Workflow

1. Restate the project goal and operating context.
2. Split requirements into functional and non-functional groups.
3. Identify assumptions and unresolved questions.
4. Define acceptance criteria for each major requirement.
5. Highlight risks that affect implementation, QA, security, or release.
6. Recommend the smallest next design step.

## Expected Output

- Requirements inventory
- Acceptance criteria
- Assumptions
- Risks
- Open questions
- Recommended next step

## Validation

- Each requirement has a clear source or rationale.
- Acceptance criteria are testable.
- Open questions distinguish blockers from non-blocking unknowns.
